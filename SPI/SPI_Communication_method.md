# 🛜 **SPI 통신 방식 (구체적)**

```C
static HAL_StatusTypeDef ICM20948_WriteReg(ICM20948_Handle_t *hicm, uint8_t reg, uint8_t data)
{
  uint8_t buf[2] = { reg, data };											// 배열로 2개를 묶음
  HAL_GPIO_WritePin(hicm->cs_port, hicm->cs_pin, GPIO_PIN_RESET);			// LOW로 둠으로 써 CS핀 해제 CS핀 읽기모드로 하겠다라는 뜻
  HAL_StatusTypeDef ret = HAL_SPI_Transmit(hicm->hspi, buf, 2, 100);		// ICM 칩에 값을 보냄
  HAL_GPIO_WritePin(hicm->cs_port, hicm->cs_pin, GPIO_PIN_SET);				// CS핀을 HIGH를 함으로 써 보낼거 다 보냈으니까 더 이상 읽기 모드로 하지 않겠다라는 뜻 이 설정대로 가겠다라는 뜻
  return ret;
}
```
- 위에 보면 대표적으로 센서 칩에 값을 전달하는 SPI 통신방식이다 정확히는 STM32IDE 툴 방식이라고 할 수 있다.

```C
ICM20948_WriteReg(hicm, ICM20948_PWR_MGMT_1, 0x80);     // ICM20948_PWR_MGMT_1 = 0x06
HAL_Delay(50);
ICM20948_WriteReg(hicm, ICM20948_PWR_MGMT_1, 0x01);     // ICM20948_PWR_MGMT_1 = 0x06
HAL_Delay(10);
ICM20948_WriteReg(hicm, ICM20948_PWR_MGMT_2, 0x00);     //  ICM20948_PWR_MGMT_2 = 0x07
```
- 위에 함수를 호출해서 사용을 한것이다
- 지금 보면 ICM센서에 대해서 초기 설정을 하고 있는 것인데 PWR1을 설정을 하고 있는것이다

- 여기서 보면 저기 있는 레지스터 주소값이 WriteReg 함수에 들어가서 배열로 들어가게되는데
- 그리고 `HAL_SPI_Transmit();` 로 ICM칩 센서에 값을 전달을 하게 된다.

## **val은 배열(포인터)이 아닙니다**
`HAL_SPI_TransmitReceive` 함수의 두 번째와 세 번째 인자는 데이터를 주고받을 **메모리의 주소(포인터)**를 원합니다.
- **현재 코드:** `uint8_t val = 0;` 이라고 선언하고 함수에 그냥 `val`을 넣으셨죠?
- **문제점:** 이렇게 하면 컴파일러는 `val`의 값인 0을 메모리 주소 `0x00000000`으로 해석하려고 시도합니다. 프로그램이 멈추거나(HardFault) 엉뚱한 곳을 참조하게 됩니다.
- **해결책:** `&val` 처럼 주소 연산자를 붙이거나, `val`을 크기가 1인 배열로 선언해야 합니다.
    - **배열 크기 맞추기:** 2바이트로 값을 보냈으면 2바이트로 받는 것이 기본입니다. 받는 배열도 2칸으로 만드세요.
    - 보내는 값 == 받는 값 == 2바이트 이런식으로 근데 예를 들어서 보내는 값이 1바이트다 그러면 받는값도 1바이트

## 🔁 **PWR_MGMT_1 (0x06) 레지스터의 비트별 의미**
이 레지스터는 8개의 스위치(비트)로 구성되어 있습니다. `0x01`을 이진수로 바꾸면 `0000 0001`이죠? 각 자리가 무슨 역할을 하는지 데이터시트를 기준으로 살펴볼게요.

| **비트 번호** | **이름** | **값 (0x01 기준)** | **설명** |
| :--- | :--- | :--- | :--- |
| **Bit 7** | DEVICE_RESET | 0 | 1이면 칩 전체를 초기화합니다. 여기선 0이므로 리셋 안 함. |
| **Bit 6** | SLEEP | 0 | 1이면 잠듭니다. 여기선 0이므로 잠에서 깨어남(Wake up). |
| **Bit 5** | LP_EN | 0 | 1이면 저전력 모드. 여기선 0이므로 고성능 모드 유지. |
| **Bit 4-0** | 사용 안 함 | - | Reserved. |
| **Bit 3** | TEMP_DIS | 0 | 1이면 온도 센서 끄기. 여기선 0이므로 온도 센서 켬. |
| **Bit 2:0** | CLKSEL | 001 | 클럭 소스 선택. |

```C
0 0 0 0 0 0 0 1 == 0x01
| | | | | | | |
7 6 5 4 3 2 1 0 Bit
```

## **왜 하필 0x01 인가요? (CLKSEL의 의미)**
가장 핵심은 마지막 3비트(`001`)인 CLKSEL입니다. 센서가 데이터를 만들려면 아주 정확한 타이밍(시계)이 필요한데, 이 옵션이 그 시계를 고르는 메뉴판입니다.
- **0 (000):** 내부 20MHz 발진기 사용 (정확도가 다소 낮음).
- **1 (001):** **Auto-select (권장 설정).** 센서가 알아서 가장 좋은 시계(자이로스코프 기반 클럭 등)를 찾아 사용합니다.
- **7 (111):** 시계를 아예 멈춤 (Stop clock).

### **즉** 
`0x01`을 보낸다는 건 **"잠에서 깨어나서(Sleep=0), 가장 똑똑한 시계를 골라 잡고(CLKSEL=1), 제대로 일할 준비를 해라!"**라고 명령을 내리는 것입니다.

## **만약 다른 값을 넣는다면?**
- `0x80` **(1000 0000):** 칩을 공장 초기화 상태로 **리셋**합니다. 보통 초기화 함수 맨 처음에 한 번 실행합니다.
- `0x40` **(0100 0000):** 칩을 다시 **잠들게(Sleep)** 합니다. 배터리를 아껴야 할 때 사용하죠.

## 🤔 **어?? 그러면 그냥 한꺼번에 보내면 안돼?? 뭐하러 이렇게 나눠서 보내??**
## 💾 **칩이 물리적으로 "생각할 시간"이 필요함 (Hardware Delay)**
제일 큰 이유는 `0x80` **(Reset)** 명령 때문입니다.
- `0x80` **전송:** 칩에게 "야, 너 지금까지 하던 거 다 버리고 공장 초기화해!"라고 명령을 내린 상태입니다.
- **칩의 반응:** 칩 내부의 수만 개 트랜지스터와 레지스터가 0으로 돌아가는 데 시간이 걸립니다.
- **문제 발생:** 만약 리셋 명령(`0x80`)을 보내자마자 바로 다음 배열에 `0x01`(깨우기)을 묶어서 보내버리면, 칩이 아직 리셋 중이라서 그 뒤에 오는 명령을 무시하거나 씹어버릴 수 있습니다.

그래서 코드 사이에 `HAL_Delay(50);`을 넣어서 칩이 완전히 초기화를 끝내고        <br>
ICM: "네, 주인님. 이제 말씀하세요."라고 대답할 수 있는 여유를 주는 것입니다.

## **레지스터 주소가 연속적이지 않을 때 (Address Jump)**
SPI는 한 번 주소를 지정하면 그다음 데이터는 자동으로 **"다음 주소(+1)"**에 저장되는 **Auto-Increment** 기능을 지원합니다.
- **연속된 경우:** `PWR_MGMT_1` (0x06) 다음에 바로 `PWR_MGMT_2` (0x07)가 있죠? 이럴 땐 {`0x06`, `0x01`, `0x00`} 이렇게 3바이트를 묶어서 보내면 0x06에 0x01이, 0x07에 0x00이 순서대로 들어갑니다.
- **끊긴 경우:** 만약 0x06번지를 설정하고 갑자기 0x20번지를 설정해야 한다면? 배열로 묶어서 보낼 수가 없습니다. 중간에 비어있는 주소들에 엉뚱한 값이 들어갈 수 있기 때문입니다.

## **"그럼 리셋 말고 일반 설정은 묶어도 되나요?"**
👉 **네! 리셋 과정이 끝난 뒤라면 가능합니다.**

예를 들어, 리셋이 다 끝난 후에 `PWR_MGMT_1`과 `PWR_MGMT_2`를 동시에 설정하고 싶다면 아래처럼 짤 수 있습니다.

```C
// 0x06번지부터 시작해서 2바이트를 연달아 쓰겠다!
uint8_t setup_buf[3] = { 0x06, 0x01, 0x00 }; 

HAL_GPIO_WritePin(cs_port, cs_pin, GPIO_PIN_RESET);
HAL_SPI_Transmit(&hspi1, setup_buf, 3, 100); // [주소0x06] [값0x01] [값0x00]
HAL_GPIO_WritePin(cs_port, cs_pin, GPIO_PIN_SET);
```
이렇게 하면 칩은 첫 번째 값(`0x01`)은 `0x06`에 넣고, 두 번째 값(`0x00`)은 자동으로 그다음 주소인 `0x07`에 넣어줍니다.


## 💡 **요약하자면**
- **리셋(0x80)**은 칩이 정신 차릴 시간이 필요하므로 **무조건 단독으로 보내고 기다려줘야** 합니다.
- 그 외에 **주소가 붙어있는 레지스터들**은 배열로 묶어서 한 번에 쏘는 것이 통신 속도 면에서 훨씬 유리합니다. (이걸 **Burst Write**라고 부릅니다.)

👉 **쉽게말해서 리셋만 아니면 다른 설정들은 다 묶어서 사용이 가능하다**
#### **예시:**
`PWR_MGMT_1` 레지스터(주소 `0x06`)에는 여러 기능이 섞여 있습니다.
- **Bit 6 (`SLEEP`):** 1이면 잠자기, 0이면 깨어남
- **Bit 5 (`LP_EN`):** 1이면 저전력 모드
- **Bit 2:0 (`CLKSEL`):** 클럭 소스 선택 (001: Auto-select)

만약 **"칩을 깨우고(Sleep 해제), 자동 클럭을 사용하겠다"**는 설정을 하고 싶다면?
- `SLEEP` (0) + `CLKSEL` (001) = `0x01`
- 이건 한 번에 `ICM20948_WriteReg(hicm, 0x06, 0x01);` 로 보내도 아무 문제 없습니다. 칩이 깨어나면서 동시에 클럭 설정을 받아들입니다.

지금 작성하신 코드는 센서를 안전하게 깨우기 위한 **'가장 보수적이고 안전한 방식'**이라고 보시면 됩니다.

## 🧠 **추가적인 OpenAI 설명**
## MSB의 역할: "명령의 성격" 결정
칩 내부에는 Write 회로와 Read 회로가 따로 있다고 생각하면 쉽습니다.
- **MSB = 0 (Write 모드):** 칩이 "아, 지금 들어오는 데이터들을 내 레지스터에 **저장(Overwriting)** 하라는 거구나!"라고 판단합니다. 이때 비로소 센서의 설정값이 바뀝니다.
- **MSB = 1 (Read 모드):** 칩이 "아, 주인이 내 안에 뭐가 들었는지 **구경(View)** 만 하겠다는 거구나!"라고 판단합니다. 이때는 내부 회로가 '쓰기'를 차단하고, 요청한 주소의 값만 복사해서 밖으로 내보냅니다.

## 질문에 대한 답변: "읽기면 설정이 안 되나요?"
**네, 맞습니다. 설정이 절대 되지 않습니다.**
- 만약 `0x06 | 0x80` (즉, `0x86`)을 주소로 보내면, 칩은 **"나한테 뭘 쓰려고 하지 말고, 네가 궁금한 0x06번지 값을 나한테 보여주기나 해!"**라고 인식합니다.
- 따라서 그 뒤에 아무리 `0x80`이라는 데이터를 보내도, 칩은 그걸 저장하지 않고 그냥 무시해버립니다.
- 결과적으로 PWR1 **레지스터의 설정값은 변하지 않습니다.**

## 왜 이렇게 복잡하게 만들었을까요?
만약 이런 구분이 없다면, 우리가 단순히 값을 **확인(Read)**만 하려고 할 때도 의도치 않게 센서 설정이 꼬여버릴 수 있기 때문입니다.
- **설정할 때 (Write):** 주소(`0x06`) + 데이터(`0x80`) -> **센서 동작 변경 O**
- **확인할 때 (Read):** 주소(`0x86`) + 더미(`0x00`) -> **센서 동작 변경 X** (단순 조회)

## 실전 팁: 제대로 설정됐는지 확인하는 법
보통 임베디드 개발자들은 아래와 같은 순서로 코딩합니다.
- **Write:** `0x06` 주소에 `0x80`을 써서 센서를 깨웁니다.
- **Read:** `0x86` 주소를 불러서 정말로 `0x80`이 들어있는지 읽어옵니다.
- **Compare:** 내가 쓴 값(`0x80`)과 읽어온 값(`0x80`)이 같은지 비교해서 "성공!"을 띄웁니다.

<br>

## **동작 과정**
1. `HAL_GPIO_WritePin(..., RESET):` 칩에게 "야, 이제 너한테 말 걸 거야. 준비해!"라고 신호를 줍니다.

2. `HAL_SPI_Transmit(..., buf, 2, ...):`
    - **Step A:** `buf[0]` (레지스터 주소)이 나갑니다. ICM 칩 내부의 로직이 이 주소를 낚아채서 "Write 대기 상태"가 됩니다.
    - **Step B:** `buf[1]` (실제 데이터)이 나갑니다. 칩은 방금 대기시킨 주소 칸에 이 데이터를 쏙 집어넣습니다.

3. `HAL_GPIO_WritePin(..., SET):` "이제 끝났어. 쉬어."라고 하며 통신을 종료합니다.

## **왜 배열로 묶어서 보내나요?**
`HAL_SPI_Transmit` 함수를 두 번 나눠서 호출해도 되지만, 배열로 묶어서 한 번에 보내는 이유가 있습니다.
- **연속성:** SPI는 클럭 기반 통신이라 데이터 사이에 공백이 생기면 칩이 오작동할 가능성이 있습니다. (칩에 따라 다름)
- **효율성:** HAL 함수를 호출하는 오버헤드를 줄여서 통신 속도를 최적화할 수 있습니다.

## 💡 **주의할 점 (꿀팁)**
ICM20948 같은 센서는 **Read**를 할 때와 **Write**를 할 때 레지스터 주소값의 MSB(최상위 비트)를 다르게 설정해야 할 수도 있습니다.
- **Write 시:** `reg` 주소 그대로 사용 (보통 MSB가 0)
- **Read 시:** `reg | 0x80` (MSB를 1로 만듦)

## 🤔 **왜 꼭 2바이트로 보내야하는거야??**
## **왜 0x06 | 0x80을 보내고 나서 '또' 뭔가를 보내나요? (Read의 경우)**
칩(ICM20948) 입장에서 생각해보세요.
- **1바이트째 (MCU가 `0x86` 보냄):** 칩은 "아! 0x06번지 데이터를 읽으라고? 알았어, 잠깐만 기다려봐. 내가 내부 메모리에서 그 값을 꺼내올게!" 라고 판단합니다. 하지만 아직 값을 꺼내서 MCU로 보내줄 시간이 없습니다. (방금 주소를 받았으니까요!)

- **2바이트째 (MCU가 `0x00` 같은 아무 값이나 보냄):** SPI는 MCU가 클럭(박자)을 쳐줘야만 칩이 데이터를 보낼 수 있습니다. 그래서 MCU가 의미 없는 값(0x00)이라도 하나 더 보내주면서 **"야, 내가 박자 쳐줄 테니까 아까 준비한 데이터 지금 보내!"**라고 신호를 주는 겁니다.

**비유하자면:** > "1바이트째" 친구에게 "야, 너 가방에 든 것 좀 보여줘(주소 전송)"라고 말합니다.             <br>
"2바이트쨰" 친구가 가방을 여는 동안(칩 내부 처리), 당신은 손을 내밀어(추가 클럭 전송) 물건을 받아야 합니다.         <br>
손을 내밀지 않으면 물건을 받을 수 없죠? 그 '손을 내미는 행위'가 두 번째 바이트를 보내는 것입니다.

## **Write(쓰기)는 꼭 배열 2개를 묶어야 하나요?**
네, 쓰기는 **"어디에(주소) + 무엇을(데이터)"** 쓸지가 명확해야 하기 때문에 최소 2바이트가 한 세트입니다.
- **1바이트째:** 칩에게 "0x06번지에 쓸 거야!"라고 주소만 알려줌.
- **2바이트째:** "거기에 0x80이라는 값을 넣을 거야!"라고 실제 값을 전달함.

이걸 배열로 묶어서 `HAL_SPI_Transmit` 한 번에 보내는 게 가장 깔끔하고 오류가 적습니다. (따로따로 보내도 되지만, 그사이에 CS 핀이 올라가 버리면 칩이 명령을 취소해버립니다.)

## **"그냥 포인터 하나만 보내면 안 되나?"에 대한 답변**
컴퓨터 메모리처럼 주소만 틱 던지면 값이 툭 튀어나오는 게 아닙니다.
- **하드웨어적 한계:** SPI는 전선(MOSI, MISO)을 통해 비트 하나하나를 밀어내는 방식입니다.
- **약속(Protocol):** ICM 칩 설계자가 "우리 칩은 첫 번째 바이트에 주소를 받고, 두 번째 바이트 때 데이터를 내보내거나 받겠다"라고 규칙을 정해놨습니다. 우리는 그 규칙(박자)을 따라야만 합니다.

## ✅ **정리하자면 이렇습니다!**

| **동작** | **1바이트 (MCU 전송)** | **2바이트 (MCU 전송)** | **칩의 반응** |
| :--- | :--- | :--- | :--- |
| **Write** | `0x06` (주소) | `0x80` (실제 저장할 값) | `0x06`에 `0x80`을 저장함 |
| **Read** | `0x86` (주소+읽기비트) | `0x00` (아무거나/Dummy) | 2바이트째에 저장된 값을 MCU로 밀어줌 |
